Lecture 2.3: Consensus without identity: the block chain

Different from traditional distributed system, Bitcoin nodes does not have any persistent long-term identities.

If nodes did have identities

it would make things a lot easier for a couple of reasons:

  • Pragmatic: some protocols need node IDs;
  • Security: assume less than 50% malicious(If nodes were identified, they won’t able to create new node identities, then we can derive the assumption above);


Why don’t Bitcoin nodes have identities?

  • Indentity is hard in a P2P system — Sybil attack(a lot of different participants are controlled by the same adversary);
  • Pseudonymity is a goal of Bitcoin;


Weaker assumption: to select random node

  • Analogy: lottery or raffle;
  • When tracking and verifying identities is hard, we give people tokens, tickets, etc.
  • Now we can pick a random ID to select that node;

What the assumption says is that there is some ability, somehow, enable us to pick a random token ID and call upon that person.


Key idea: implicit consensus

In each round, random node is picked;

This node proposes the next block in the chain;

Other nodes implicitly accept/reject this block by:

  • extending it
  • ignoring it and extending chain from earlier block

Every block contains hash of the block it extends.


Consensus algorithms(simplified)

  1. New transactions are boardcase to all nodes;
  2. Each node collects new transactions into a block;
  3. In each round a random node gets to boardcast its block;
  4. Other nodes accept the block only if all transactions in it are valid(unspent, valid signatures);
  5. Nodes express their acceptance of the block by including its hash in the next block they create;


Do some trying to understand why this consensus algorithm works.

Instead of asking why this works, let’s try to ask how can a malicious adversary try to subvert this process.

What can a malicious node do?

a little bit try

Now, even if it is Alice’s turn to propose the next block in this chain, she can’t steal other user’s Bitcoins. It is because she can’t forge their signatures. So as long as the underlying crypto is solid, she’s not able to simply steal Bitcoins.

Another thing Alice might try to do is denying someone(Bob)’s transactions from the block she creates. But it’s nothing more than a little annoyance, because if Bob’s transactions don’t make it into the next block that Alice proposes, he will just wait another block.


Double spending attack

From Bob the merchant’s point of view

In general, the more confirmations your transaction gets, the higher the probability that it is end up on the long-term consensus chain.


Recap

Protection against invalid transactions is cryptographic, but enforced by consensus.

Protection against double-spending is purely by consensus.

You’re never 100% sure a transaction is in consensus branch. Guarantee is probabilistic.(But this exponential probability guarantee is pretty good, after about six transactions, there is virtually no chance that you’re gonna go wrong)
